# Security Rules

## Authentication
- All API endpoints require JWT auth except: `/api/token/`, `/api/token/refresh/`, `/api/auth/google/`
- Token blacklisting on logout via `rest_framework_simplejwt.token_blacklist`
- Never log tokens, passwords, or secrets

## Authorization
- Financial module (legacy core `persons`/`expenses`/‚Ä¶ AND the `finances/` app): `IsAdminUser` ‚Äî admin-only for both read and write. A non-staff tenant gets 403; the whole module is hidden from tenants (P1.2). `FinancialReadOnly` was removed.
- Tenant√óadmin segregation: `TenantViewSet`/`LeaseViewSet` querysets are scoped to the requesting tenant when non-staff (own record / own leases only); staff see everything. The tenant portal stays isolated under `/api/tenant/*` (`IsTenantUser`).
- Existing CRUD: `IsAuthenticatedOrReadOnly`, `IsAdminUser`, `IsOwnerOrAdmin`, `IsTenantOrAdmin`, `ReadOnlyForNonAdmin`
- `IsAdminUser` has a single canonical definition in `core.permissions` (is_staff OR is_superuser). Never import `IsAdminUser` from `rest_framework.permissions` (it covers only is_staff).
- Permission classes defined in `core/permissions.py` ‚Äî always use these, never inline permission checks
- Frontend: a non-HttpOnly `role` cookie (staff|tenant) is set on login so the Next.js middleware + `MainLayout` keep tenants out of the admin dashboard. Hide create/edit/delete UI for non-admin users (conditional rendering based on user.is_staff). The backend is the real barrier; the cookie is a UX hint.

## Data Validation
- CPF/CNPJ validation is mandatory ‚Äî use validators from `core/validators/`
- Always validate user input at serializer level
- Never trust client-side validation alone

## Sensitive Files
- Never read, log, or commit: `.env`, `.env.local`, credentials, API keys
- Never hardcode secrets ‚Äî use `python-decouple` / environment variables
- `detect-private-key` and `check-added-large-files` (maxkb=1000) pre-commit hooks block secrets/dumps ‚Äî but enforcement requires `pre-commit install` after cloning (the hooks are inert until installed)
- Never commit DB dumps/backups: they live only in `/backups/` (gitignored), generated by `scripts/backup_db.py`

## CORS
- Only allow configured origins (localhost:4000, localhost:6000)
- Never use `CORS_ALLOW_ALL_ORIGINS = True` in production

## SQL Injection
- Always use Django ORM or parameterized queries ‚Äî never raw SQL with string formatting

## Row Level Security (Supabase / Production)
- Production runs on Supabase (project `kaukiwhbmvnjjekodcmq`), which exposes the `public` schema through the PostgREST Data API to the `anon`/`authenticated` roles. This app does NOT use the Data API ‚Äî all access is via the Django backend, which connects as the `postgres` role (`rolbypassrls = true`, table owner) and therefore bypasses RLS.
- CRITICAL: **RLS must stay ENABLED on every `public` table.** With RLS enabled and no policies, the Data API roles are denied while the Django backend is unaffected. This is the intended end state ‚Äî the Supabase advisor's `rls_enabled_no_policy` (INFO) is expected and must NOT be "fixed" by adding permissive policies.
- CRITICAL: **Every new table needs RLS enabled in the same migration that creates it.** Add `migrations.RunSQL("ALTER TABLE public.<table> ENABLE ROW LEVEL SECURITY;", reverse_sql="ALTER TABLE public.<table> DISABLE ROW LEVEL SECURITY;")` ‚Äî pattern established in `core/migrations/0047_enable_row_level_security.py`. Use static SQL (no f-strings) to avoid ruff S608.
- After any DDL on prod, run the Supabase security advisor (`get_advisors type=security`) and confirm no `rls_disabled` (CRITICAL) entries appear.
- Do NOT add `FORCE ROW LEVEL SECURITY` ‚Äî it would make even the owner/`postgres` role subject to RLS and break the backend.
